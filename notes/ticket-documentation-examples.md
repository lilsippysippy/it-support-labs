# Ticket Documentation Examples

This file documents beginner ticket-style examples for IT support practice.

## Purpose

The goal of this lab is to practice writing clear IT support ticket notes. Good documentation helps other technicians understand the issue, what was tried, what resolved it, and whether anything needs to be escalated.

## Why Ticket Documentation Matters

In IT support, tickets help track:

* The user or device affected
* The issue being reported
* Troubleshooting steps taken
* Commands or tools used
* Resolution details
* Escalation notes
* Follow-up needs

## Basic Ticket Format

A clear ticket note should include:

* Issue summary
* User impact
* Troubleshooting steps
* Findings
* Resolution
* Final status

## Example 1: Password Reset

### Issue Summary

User reported they could not sign in because their password was not working.

### User Impact

User was unable to access their account and begin work.

### Troubleshooting Steps

* Verified the user according to policy
* Confirmed the user was entering the correct username
* Checked for account lockout
* Completed password reset process
* Asked user to sign in with temporary password
* Confirmed user could create a new password

### Resolution

Password was reset successfully. User confirmed they could sign in.

### Final Status

Resolved.

## Example 2: Printer Not Printing

### Issue Summary

User reported that their document was not printing.

### User Impact

User could not print required documents.

### Troubleshooting Steps

* Confirmed printer was powered on
* Checked for paper or toner issues
* Checked print queue for stuck jobs
* Restarted printer
* Checked Print Spooler service
* Sent test print

### Resolution

Cleared stuck print job and restarted printing. User confirmed printer was working.

### Final Status

Resolved.

## Example 3: Network Connectivity Issue

### Issue Summary

User reported that websites would not load even though Wi-Fi appeared connected.

### User Impact

User could not access web-based tools.

### Troubleshooting Steps

* Confirmed device was connected to Wi-Fi
* Ran `ipconfig` to check IP address information
* Ran `ping 8.8.8.8` to test internet connectivity
* Ran `ping google.com` to test DNS/name resolution
* Restarted browser
* Flushed DNS cache using `ipconfig /flushdns`

### Resolution

DNS cache was flushed and browser was restarted. User confirmed websites loaded again.

### Final Status

Resolved.

## Example 4: Microsoft 365 Access Issue

### Issue Summary

User reported they could not access a shared Microsoft 365 file.

### User Impact

User could not open a shared document needed for work.

### Troubleshooting Steps

* Confirmed user was signed into the correct account
* Confirmed the exact file and location
* Asked for the error message
* Checked whether other users could access the file
* Reviewed possible permission issue
* Documented details for escalation

### Resolution

Issue required permission review and was escalated to the appropriate administrator.

### Final Status

Escalated.

## Documentation Tips

When writing ticket notes, I should:

* Be clear and specific
* Avoid guessing
* Include error messages when available
* List troubleshooting steps in order
* Document commands or tools used
* Include the final result
* Escalate when the issue is outside my access or responsibility

## What I Am Learning

I am learning how to document IT support issues in a professional and organized way. This helps build good habits for help desk, desktop support, and production support roles.

## Safety Note

I should never include passwords, MFA codes, private user information, company data, internal system names, or sensitive screenshots in public GitHub documentation.
